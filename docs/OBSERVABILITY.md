# MarketPulse Observability Guide

## Overview

MarketPulse exposes comprehensive metrics for monitoring the entire data pipeline, including event ingestion, normalization, persistence, and publishing. All metrics are compatible with Prometheus and can be scraped at `/metrics`.

## Metrics Endpoint

**URL:** `http://localhost:8080/metrics`  
**Format:** Prometheus text format  
**Scrape Interval:** 5 seconds (configurable in `prometheus.yml`)

## Core Pipeline Metrics

### 1. Events Ingested (Feed Layer)

These metrics track raw events generated by the feed:

| Metric | Type | Description |
|--------|------|-------------|
| `feed_events_ingested_total` | Counter | Total events produced by feed |
| `feed_l1_events_total` | Counter | Level 1 (BBO) quote events |
| `feed_l2_events_total` | Counter | Level 2 (order book) events |
| `feed_trade_events_total` | Counter | Trade execution events |

**Query Examples:**
```promql
# Event ingestion rate (events/sec)
rate(feed_events_ingested_total[1m])

# Event type breakdown
rate(feed_l1_events_total[1m])
rate(feed_l2_events_total[1m])
rate(feed_trade_events_total[1m])

# Total events in last hour
increase(feed_events_ingested_total[1h])
```

### 2. Normalization Metrics

Track the conversion of RawEvent → Frame:

| Metric | Type | Description |
|--------|------|-------------|
| `normalizer_events_total` | Counter | Total events processed |
| `normalizer_errors_total` | Counter | Errors during normalization |
| `normalizer_failures_total` | Counter | Failed normalizations (alias for errors) |
| `normalizer_event_latency_ns_p50` | Histogram | P50 normalization latency (ns) |
| `normalizer_event_latency_ns_p95` | Histogram | P95 normalization latency (ns) |
| `normalizer_event_latency_ns_p99` | Histogram | P99 normalization latency (ns) |
| `normalizer_event_latency_ns_max` | Histogram | Max normalization latency (ns) |
| `normalizer_event_latency_ns_count` | Histogram | Count of latency samples |

**Query Examples:**
```promql
# Normalization throughput (events/sec)
rate(normalizer_events_total[1m])

# Error rate
rate(normalizer_errors_total[1m])

# Error percentage
100 * rate(normalizer_errors_total[1m]) / rate(normalizer_events_total[1m])

# P99 latency in microseconds
normalizer_event_latency_ns_p99 / 1000

# P95 latency in milliseconds
normalizer_event_latency_ns_p95 / 1000000
```

### 3. Records Written (Persistence Layer)

Track persistence to disk:

| Metric | Type | Description |
|--------|------|-------------|
| `recorder_frames_total` | Counter | Total frames processed by recorder |
| `recorder_records_written_total` | Counter | Total records written to MDF files |
| `recorder_fsyncs_total` | Counter | Number of fsync operations |
| `recorder_current_file_bytes` | Gauge | Current MDF file size in bytes |
| `recorder_write_frame_ns_p50` | Histogram | P50 write latency (ns) |
| `recorder_write_frame_ns_p95` | Histogram | P95 write latency (ns) |
| `recorder_write_frame_ns_p99` | Histogram | P99 write latency (ns) |

**Query Examples:**
```promql
# Write throughput (records/sec)
rate(recorder_records_written_total[1m])

# Current file size in MB
recorder_current_file_bytes / (1024 * 1024)

# Fsync frequency
rate(recorder_fsyncs_total[1m])

# P99 write latency in microseconds
recorder_write_frame_ns_p99 / 1000

# Average write latency
recorder_write_frame_ns_p50 / 1000
```

### 4. Publish Latency (Distribution Layer)

Track the latency of publishing frames to clients:

| Metric | Type | Description |
|--------|------|-------------|
| `publisher_publish_latency_ns_p50` | Histogram | P50 publish latency (ns) |
| `publisher_publish_latency_ns_p95` | Histogram | P95 publish latency (ns) |
| `publisher_publish_latency_ns_p99` | Histogram | P99 publish latency (ns) |
| `publisher_publish_latency_ns_max` | Histogram | Max publish latency (ns) |
| `publisher_frames_published_total` | Counter | Total frames published |
| `publisher_frames_dropped_backpressure` | Counter | Frames dropped due to backpressure |
| `publisher_frames_dropped_queue_full` | Counter | Frames dropped (queue full) |
| `publisher_active_clients` | Gauge | Number of active TCP clients |

**Query Examples:**
```promql
# P99 publish latency in microseconds
publisher_publish_latency_ns_p99 / 1000

# Publish rate (frames/sec)
rate(publisher_frames_published_total[1m])

# Drop rate
rate(publisher_frames_dropped_backpressure[1m])

# Active clients
publisher_active_clients

# Latency percentiles comparison
publisher_publish_latency_ns_p50 / 1000
publisher_publish_latency_ns_p95 / 1000
publisher_publish_latency_ns_p99 / 1000
```

## Pipeline Health Metrics

### Queue Depths

Monitor the health of internal queues:

| Metric | Type | Description |
|--------|------|-------------|
| `pipeline_feed_queue_approx` | Gauge | Feed → Normalizer queue depth |
| `pipeline_normalizer_to_publisher_queue_approx` | Gauge | Normalizer → Publisher queue depth |
| `pipeline_normalizer_to_recorder_queue_approx` | Gauge | Normalizer → Recorder queue depth |

**Query Examples:**
```promql
# Queue depths
pipeline_feed_queue_approx
pipeline_normalizer_to_publisher_queue_approx
pipeline_normalizer_to_recorder_queue_approx

# Queue capacity utilization (assuming 100k capacity)
100 * pipeline_feed_queue_approx / 100000
```

### Distribution Metrics

| Metric | Type | Description |
|--------|------|-------------|
| `frame_distribution_total` | Counter | Total frames distributed from normalizer |

**Query Examples:**
```promql
# Distribution rate
rate(frame_distribution_total[1m])
```

## Prometheus Configuration

### Scrape Configuration

The system is pre-configured to be scraped by Prometheus. See `infra/prometheus/prometheus.yml`:

```yaml
scrape_configs:
  - job_name: 'market-data-core'
    static_configs:
      - targets: ['md-core:8080']
    metrics_path: '/metrics'
    scrape_interval: 5s
```

### Running Prometheus

#### With Docker Compose

```bash
# Start entire stack (including Prometheus)
docker-compose up -d

# Access Prometheus UI
open http://localhost:9090
```

#### Standalone Prometheus

```bash
# Using Docker
docker run -d \
  -p 9090:9090 \
  -v $(pwd)/infra/prometheus/prometheus.yml:/etc/prometheus/prometheus.yml \
  prom/prometheus

# Or native binary
prometheus --config.file=infra/prometheus/prometheus.yml
```

## Key Monitoring Queries

### System Health Check

```promql
# Is data flowing?
rate(feed_events_ingested_total[1m]) > 0
rate(normalizer_events_total[1m]) > 0
rate(recorder_records_written_total[1m]) > 0
rate(publisher_frames_published_total[1m]) > 0

# Are there errors?
rate(normalizer_errors_total[5m]) > 0

# Are queues backing up? (alert if > 50k)
pipeline_feed_queue_approx > 50000
pipeline_normalizer_to_publisher_queue_approx > 50000
pipeline_normalizer_to_recorder_queue_approx > 50000
```

### Performance Monitoring

```promql
# End-to-end latency (aggregate percentiles)
histogram_quantile(0.99, 
  rate(normalizer_event_latency_ns_bucket[5m])
) / 1000

histogram_quantile(0.99, 
  rate(publisher_publish_latency_ns_bucket[5m])
) / 1000

# Throughput comparison
rate(feed_events_ingested_total[1m])
rate(normalizer_events_total[1m])
rate(recorder_records_written_total[1m])
rate(publisher_frames_published_total[1m])
```

### Error Monitoring

```promql
# Error rate per minute
rate(normalizer_errors_total[1m])

# Error percentage
100 * (
  rate(normalizer_errors_total[5m]) / 
  rate(normalizer_events_total[5m])
)

# Frames dropped
rate(publisher_frames_dropped_backpressure[1m])
rate(publisher_frames_dropped_queue_full[1m])
```

## Alert Rules

### Recommended Alerts

Create alerts in Prometheus or Grafana:

```yaml
groups:
  - name: marketpulse
    interval: 30s
    rules:
      # Pipeline stalled
      - alert: PipelineStalled
        expr: rate(feed_events_ingested_total[2m]) == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Pipeline has stalled - no events ingested"
          
      # High error rate
      - alert: HighNormalizationErrors
        expr: rate(normalizer_errors_total[5m]) > 100
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "High normalization error rate: {{ $value }}/sec"
          
      # High latency
      - alert: HighPublishLatency
        expr: publisher_publish_latency_ns_p99 > 10000000
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "P99 publish latency > 10ms: {{ $value }}ns"
          
      # Queue backpressure
      - alert: QueueBackpressure
        expr: pipeline_feed_queue_approx > 80000
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "Feed queue backing up: {{ $value }} items"
          
      # Recorder falling behind
      - alert: RecorderBehind
        expr: |
          rate(normalizer_events_total[1m]) - 
          rate(recorder_records_written_total[1m]) > 10000
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Recorder falling behind normalizer"
```

## Metric Naming Conventions

MarketPulse follows Prometheus naming best practices:

- **Counters:** `_total` suffix (monotonically increasing)
- **Gauges:** No suffix (can go up or down)
- **Histograms:** `_ns` suffix with percentile labels

**Examples:**
- ✅ `feed_events_ingested_total` (counter)
- ✅ `publisher_active_clients` (gauge)
- ✅ `normalizer_event_latency_ns_p99` (histogram)

## Common Troubleshooting

### Metrics Not Appearing

1. **Check endpoint:**
   ```bash
   curl http://localhost:8080/metrics
   ```

2. **Check Prometheus targets:**
   Visit `http://localhost:9090/targets`
   
3. **Verify scrape config:**
   ```bash
   cat infra/prometheus/prometheus.yml
   ```

### Stale Metrics

- Counters always increase (never reset during runtime)
- Gauges reflect current state
- Histograms accumulate over time

To reset: Restart MarketPulse

### High Latency Values

Histogram buckets are in nanoseconds:
- 1,000 ns = 1 µs
- 1,000,000 ns = 1 ms
- 10,000,000 ns = 10 ms
